# Lessons

- This project uses Xcode's filesystem-synchronized root group, so new source files under `PlayStatus/` compile automatically. Do not manually patch `project.pbxproj` unless the synced-group model truly requires an exception.
- Walkthrough-style windows in this app must treat the 300 px sidebar as a hard layout cost. Any preview-heavy two-column step needs an adaptive stack fallback and scrolling, or the preview card can compress the instructional content into blank space.
- When a walkthrough mixes a dark window chrome with bright glass cards, light-surface sections must opt into a light color scheme or explicit dark ink colors. Relying on `.primary` and `.secondary` in dark mode makes the setup content unreadable even when the layout is correct.
- PlayStatus now targets macOS 15, so walkthrough-specific UI can use newer SwiftUI animation/background APIs directly, but upgrading the floor will surface deprecation warnings in older view code that should be cleaned up separately.
- Multi-step ScrollViews keep their previous offset unless you reset them explicitly. For guided walkthroughs, always scroll back to a stable top anchor on step changes and add hover/selection feedback to plain-button surfaces so navigation looks interactive.
- Coachmarks in narrow sidebars should not be implemented as overlays on top of the controls they describe. Prefer inline placement or reserved layout space so the tip teaches the UI without blocking it.
- Walkthrough and demo surfaces must never observe the shared live playback model directly. Use draft/session state plus cached preview assets so provider polling and marquee updates do not invalidate the onboarding UI.
- In the SwiftUI app lifecycle, walkthrough or onboarding buttons should open Settings through the `openSettings` environment action, not the older AppKit `showSettingsWindow:` selector. The selector path can silently fail even when a SwiftUI `Settings` scene exists.
- When a walkthrough uses draft state, do not synchronously write that draft back into the shared model before changing steps. Advance the UI first, then defer refresh-triggering side effects like provider updates or launch-at-login registration to the next main-actor turn.
- Do not wrap walkthrough-wide step changes in `withAnimation` when expensive step-dependent surfaces like `MeshGradient` backdrops are present. Animate the content pane locally and disable animation on the backdrop, or the whole window can hitch on page changes.
- When replacing a sluggish walkthrough, remove the old implementation completely instead of leaving duplicate step or backdrop types in the file. A single stable shell plus lightweight early-step surfaces is much easier to reason about, profile, and keep fast.
- Settings-oriented walkthrough previews should look like preset summaries, not feature-tour hero cards. If a step is about defaults and personalization, use balanced summary tiles and one restrained visual anchor instead of dropping in a large player mockup that steals the layout.
- For codebase-reduction work in PlayStatus, track two outcomes separately: raw LOC and compile-shape complexity. Shared helper extractions can eliminate SwiftUI type-check hotspots and shrink monoliths even when total LOC stays roughly flat, so the review should explicitly report both metrics instead of treating them as the same win.
